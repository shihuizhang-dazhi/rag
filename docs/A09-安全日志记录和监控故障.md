# OWASP Top 10: A09 - 安全日志记录和监控故障 (Security Logging and Monitoring Failures)

## 概述

安全日志记录和监控故障（原称不足的日志记录和监控）指无法检测、响应和从安全事件中恢复。没有充分的日志记录和监控，攻击者可以长期潜伏在系统中而不被发现，导致更严重的数据泄露和系统破坏。

## 常见场景

### 1. 日志记录不足
- 登录失败未记录
- 重要操作无审计日志（删除、修改、导出）
- 高危交易未打日志
- 日志中缺少时间戳、IP、用户 ID 等关键字段

### 2. 日志存储不安全
- 日志写入本地文件，易被删除
- 无日志备份机制
- 日志无访问控制
- 日志中记录敏感数据（密码、Token）

### 3. 监控缺失
- 无实时告警机制
- 无法检测暴力破解
- 未监控异常流量模式
- 缺乏自动化响应

### 4. 日志不可审计
- 日志格式不一致
- 无集中日志管理
- 日志保留时间过短
- 无法关联分析

### 5. 响应流程缺失
- 发现事件后无标准处理流程
- 无升级机制
- 无事件复盘流程
- 未进行渗透测试和红蓝演练

## 应该记录的关键事件

| 事件类型 | 记录内容 | 级别 |
|---------|---------|------|
| 登录成功/失败 | 用户名、IP、时间、UserAgent | INFO/WARN |
| 权限变更 | 操作人、目标、变更内容 | WARN |
| 数据导出 | 用户、数据量、导出范围 | WARN |
| 敏感操作 | 删除、批量修改 | WARN |
| 配置变更 | 变更项、旧值、新值 | WARN |
| 异常活动 | 罕见时间段访问、大量请求 | ERROR |
| API 错误 | 端点、错误信息、请求来源 | ERROR |

## 日志体系建设

### ELK Stack 方案
```yaml
# Filebeat 采集应用日志
filebeat.inputs:
- type: log
  paths:
    - /var/log/app/*.log
  fields:
    app: web-application

output.elasticsearch:
  hosts: ["elasticsearch:9200"]
```

### 日志格式规范
```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "level": "WARN",
  "event": "LOGIN_FAILED",
  "user": "admin",
  "ip": "192.168.1.100",
  "user_agent": "Mozilla/5.0...",
  "reason": "invalid_password",
  "attempt": 5,
  "request_id": "abc-123-def"
}
```

### 告警规则示例
```yaml
# 暴力破解检测
alert: BruteForce
expr: rate(login_failed_total[5m]) > 10
for: 5m
labels:
  severity: critical
annotations:
  summary: "检测到暴力破解攻击"
```

## 防御措施

- **统一日志格式**：所有应用使用相同的日志格式和字段
- **集中日志管理**：使用 ELK、Splunk 等集中收集和分析
- **实时监控告警**：关键指标设置告警阈值，自动化通知
- **保留策略**：日志保留不少于 90 天，敏感事件日志保留更久
- **访问控制**：日志系统需要独立认证和授权
- **定期演练**：模拟攻击事件，检验检测和响应能力
- **WAF/SIEM 集成**：实现自动化攻击检测和阻断
- **完整性保护**：日志使用追加写入或区块链等防篡改技术

## MTTR 指标

平均修复时间（Mean Time to Repair）是衡量安全运维能力的关键指标：
- MTTD（平均检测时间）：从事件发生到发现
- MTTR（平均响应时间）：从发现到遏制
- MTTC（平均止损时间）：从遏制到完全恢复

业界平均：MTTD 约 200 天，目标应缩短到小时级别。
